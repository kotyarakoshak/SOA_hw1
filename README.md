# Marketplace: архитектурное проектирование (C4 + Docker)

Это репозиторий для задание: спроектировать архитектуру маркетплейса, описать её C4 Container диаграммой (likec4) и поднять 1 сервис в Docker, который отвечает `200 OK` на `/health`.

## 1) Домены и ответственность

**1. Users (Пользователи)**
- покупатели и продавцы, профили
- роли/права (RBAC), учетные записи

**2. Catalog (Каталог)**
- товары, категории, атрибуты, изображения (метаданные)
- цены и наличие (упрощённо)
- публикация изменений товара

**3. Feed/Recommendations (Лента/персонализация)**
- формирование персонализированной выдачи
- хранение сигналов (просмотры, клики, покупки) и фичей
- деградация: если данных мало — популярное/новинки/категории

**4. Orders (Заказы)**
- создание заказа, позиции, статусы (Created -> Paid -> Shipped -> Delivered/Cancelled)
- идемпотентность (по ключу запроса) как архитектурный принцип

**5. Payments (Платежи)**
- учёт платежей, транзакций
- интеграция с платёжным провайдером (PSP) через адаптер

**6. Notifications (Уведомления)**
- отправка уведомлений о статусах заказа/платежа
- каналы: email/sms/push (в диаграмме абстрактно)

## 2) Разбиение на сервисы (логика)

- `User Service` — изолирует данные пользователей и безопасность
- `Catalog Service` — отвечает за товарные данные и их целостность
- `Feed Service` — отдельно, потому что рекомендации развиваются независимо и часто требуют другого хранения/кэшей
- `Order Service` — отдельный жизненный цикл и строгая целостность данных заказа
- `Payment Service` — отделение из-за интеграций, рисков и требований аудита
- `Notification Service` — асинхронный воркер; не должен тормозить оформление заказа

Дополнительно:
- `API Gateway / BFF` — единая точка входа для клиента; упрощает клиент и скрывает внутреннюю топологию

## 3) Владение данными (границы)

Важно: **нет общей БД**.

- User Service владеет `UserDB`
- Catalog Service владеет `CatalogDB` (+ индекс поиска)
- Feed Service владеет `FeedStore`
- Order Service владеет `OrderDB`
- Payment Service владеет `PaymentDB`
- Notification Service владеет `NotifDB`

## 4) Взаимодействия (sync/async)

**Синхронно (REST/HTTP)**:
- Клиент -> API Gateway
- API Gateway -> доменные сервисы (User/Catalog/Feed/Order/Payment)

**Асинхронно (события через брокер)**:
- Order Service публикует: `OrderCreated`, `OrderStatusChanged`
- Payment Service публикует: `PaymentSucceeded/Failed`
- Catalog Service публикует: `ProductChanged`
- Notification Service подписан на события статусов и платежей
- Feed Service подписан на события для персонализации (поведенческие сигналы)

## 6) Реализованный сервис (требование Docker + /health)

В репозитории есть минимальный сервис без бизнес-логики: `marketplace-health-service`.

- Endpoint: `GET /health` -> `200 OK`
- Технология: FastAPI + Uvicorn

### Запуск через Docker Compose

```bash
docker compose up --build
```

Проверка:
```bash
curl -i http://localhost:8080/health
```

Остановка:
```bash
docker compose down
```

## 7) Почему так (краткая аргументация)

- **Изоляция доменов** снижает связность и упрощает развитие командой.
- **Событийная интеграция** для уведомлений/персонализации уменьшает задержки критического пути оформления заказа.
- **Отдельное владение данными** уменьшает риски “shared DB”, упрощает масштабирование и ответственность.
- **BFF/Gateway** позволяет менять внутренние сервисы без перелопачивания клиентов.
